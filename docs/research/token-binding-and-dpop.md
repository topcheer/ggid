# Token Binding and Sender-Constrained Tokens

> **Scope:** Comparative analysis of three sender-constrained token approaches.
> For DPoP implementation details, see [dpop-rfc9449.md](./dpop-rfc9449.md) (2,943 lines).

**Status:** Draft | **RFCs:** 8471 (deprecated), 9449, 8705

---

## 1. The Token Binding Problem

OAuth 2.0 access tokens are **bearer tokens**: whoever holds the token can use it.
If an attacker intercepts or steals a token (via logs, XSS, network MITM, leaked
browser storage), they can replay it until it expires. No proof of identity is
required beyond possession.

**Sender-constrained tokens** solve this by binding the token to a cryptographic
key that only the legitimate client possesses. The server checks that the caller
can prove ownership of the bound key on every request. A stolen token is useless
without the corresponding private key.

Three standards have addressed this problem:

| # | Protocol | RFC | Status |
|---|----------|-----|--------|
| 1 | Token Binding | 8471 / 8472 / 8473 | **Deprecated** |
| 2 | DPoP (Demonstrating Proof-of-Possession) | 9449 | **Active** |
| 3 | mTLS Bound Tokens | 8705 | **Active** |

The core concept in all three is the **`cnf` (confirmation) claim** defined in
RFC 7800 — it embeds a key reference inside the JWT so the resource server knows
*which* key to expect proof from.

---

## 2. Token Binding Protocol (RFC 8471 / 8472 / 8473) — DEPRECATED

### How It Worked

Token Binding operated at the **TLS layer**. A TLS extension carried the
client's Token Binding ID (TBID), derived from an ECDSA key pair generated by
the browser. The server extracted the TBID from the TLS handshake and verified
that each security token's binding matched the current TLS connection.

**Key properties:**
- Binding was **transparent to applications** — no app-layer code needed
- Provided a stable **Token Binding ID** spanning multiple TLS sessions
- Used Provided/Referred Token Binding IDs to prevent token export and replay

### Why It Was Deprecated

1. **Required TLS-layer changes**: Both browser and server had to implement the
   TLS extension — a deep protocol modification touching OpenSSL, Go TLS, nginx,
   and every HTTP library.
2. **Browser support evaporated**: Chrome flagged Token Binding as experimental
   and ultimately removed support. The IETF killed the working group in 2020.
   Firefox and Safari never shipped it.
3. **Privacy concerns**: TBIDs were origin-specific but still created a tracking
   surface.
4. **TLS 1.3 incompatibility**: The extension design was built for TLS 1.2 and
   didn't translate cleanly to TLS 1.3's encrypted handshake.

### Lesson Learned

> Token binding at the TLS layer is **too invasive**. It requires changes across
> the entire stack (browsers, servers, proxies, load balancers) and becomes a
> coordination problem no single party can solve. Successful sender-constraint
> protocols operate at the **application layer**.

**GGID: Do not implement Token Binding. It is dead.**

## 3. DPoP (RFC 9449) — Current Standard

### Mechanism

DPoP binds tokens at the **application layer** using a per-request JWT proof.
The client generates an asymmetric key pair (typically EC P-256 or RSA) and
sends a signed `DPoP` header with every request. The access token contains a
`cnf.jkt` claim — the JWK Thumbprint (RFC 7638) of the client's public key.

**The DPoP proof JWT contains:**
- `htm`: HTTP method | `htu`: HTTP URI (no query string)
- `jti`: Unique nonce (replay protection) | `iat`: Timestamp (freshness)
- `ath`: Access token hash (for protected resource requests)

The resource server verifies the proof JWT signature, checks `jkt == cnf.jkt`,
validates `htm`/`htu` match the request, and tracks `jti` for replay prevention.

### Pros

- **Works in browsers** via the WebCrypto API (generate EC key, sign JWT)
- **No PKI required** — clients self-generate keys; no certificate authority
- **No TLS changes** — operates entirely in HTTP headers
- **No certificate management** — keys are non-exportable in WebCrypto
- Works for SPAs, mobile apps, and server-side clients alike

### Cons

- **One JWT per request**: Each API call requires generating and verifying a
  signature (ECDSA verify ~0.1ms, but adds up at scale)
- **Application-layer only**: Does not protect the TLS channel itself
- **Replay window**: The `jti` nonce must be tracked server-side (Redis, short TTL)
- **Key management**: Client must persist the private key across sessions
  (IndexedDB for browsers, Keychain/Keystore for mobile)

### Implementation Effort

~2 weeks for production-ready implementation. See
[dpop-rfc9449.md](./dpop-rfc9449.md) for the full guide, including nonce
challenges, key thumbprint computation, and the `DPoP-Nonce` anti-preplay
mechanism.

## 4. mTLS Bound Tokens (RFC 8705)

### Mechanism

RFC 8705 leverages **mutual TLS** — the client authenticates with an X.509
certificate during the TLS handshake. The access token is bound to the
certificate's SHA-256 thumbprint via the `cnf.x5t#S256` claim.

Two modes: **PKI-Bound Tokens** (token bound to cert thumbprint directly) and
**TLS Session-Bound Tokens** (token bound to TLS session ID). PKI-bound is more
commonly deployed.

The resource server extracts the client cert from the TLS handshake (or from a
forwarded `X-Client-Cert` header behind a proxy), computes its SHA-256
thumbprint, and verifies it matches the `cnf.x5t#S256` claim.

### Pros

- **Strongest security**: Client certificates can be hardware-backed
  (TPM, HSM, Secure Enclave) — private keys never leave the device
- **No application-layer changes**: The `Authorization: Bearer` header works
  as-is; verification happens in the TLS layer
- **No per-request signature overhead**: Cert verified once during TLS handshake
- **Mature ecosystem**: mTLS is widely deployed in service meshes (Istio,
  Linkerd) and Kubernetes

### Cons

- **Requires PKI infrastructure**: A CA must issue, rotate, and revoke client
  certificates — significant operational burden
- **Browser UX is poor**: Browsers show a cert selection popup; no JS control
- **Not suitable for SPAs**: WebCrypto cannot present client certificates
- **Cert rotation complexity**: Expiry requires coordination across client, CA,
  and token issuance
- **Proxy limitations**: TLS termination at a load balancer breaks mTLS unless
  `X-Client-Cert` header forwarding is configured

### Best For

Service-to-service communication, backend API clients, enterprise/government
systems where PKI is established, and IoT devices with hardware security modules.

### Implementation Effort

~1 week: Configure Go TLS server with `tls.RequireAnyClientCert`, extract client
cert thumbprint, verify `cnf.x5t#S256` in gateway middleware.

## 5. Comparison Table

| Feature | Token Binding (RFC 8471) | DPoP (RFC 9449) | mTLS (RFC 8705) |
|---|---|---|---|
| **Mechanism** | TLS extension + TBID | JWT proof per request | Client cert thumbprint |
| **Layer** | TLS | Application (HTTP header) | TLS |
| **cnf claim** | N/A | `cnf.jkt` (JWK thumbprint) | `cnf.x5t#S256` (cert thumb) |
| **Browser support** | Removed (~2020) | WebCrypto API | Poor (cert popup, no JS) |
| **SPA support** | No | Yes | No |
| **Mobile support** | No | Yes (native crypto) | Yes (if cert provisioned) |
| **Certificate needed** | No | No (self-gen key pair) | Yes (X.509 from CA) |
| **PKI required** | No | No | Yes |
| **Per-request overhead** | None | ECDSA sign+verify (~0.2ms) | None (TLS only) |
| **Replay protection** | Implicit (TLS channel) | `jti` nonce tracking | Implicit (TLS cert) |
| **Security level** | High (TLS-bound) | Medium-High (app-layer) | Highest (HW-backed certs) |
| **Deployment complexity** | Very High | Medium | High (PKI lifecycle) |
| **Proxy/CDN friendly** | No | Yes | Requires cert forwarding |
| **Current status** | **Deprecated** | **Active, widely adopted** | **Active, enterprise** |
| **Best for** | (dead) | Web apps, SPAs, mobile | Service-to-service, enterprise |

## 6. When to Use Each

| Client Type | Recommendation | Rationale |
|---|---|---|
| **SPA (browser)** | DPoP | mTLS impossible from JavaScript |
| **Web app (server)** | DPoP | No PKI needed; works with HTTP stack |
| **Mobile** | DPoP or mTLS | DPoP via native crypto; mTLS if certs via MDM |
| **Service-to-service** | mTLS | Certs common in microservices; no overhead |
| **Gateway-internal** | mTLS | Service mesh already uses mTLS |
| **IoT / Embedded** | mTLS | Hardware-backed certs (TPM/HSM) |
| **Admin / CLI** | DPoP | Easier key management; no CA dependency |
| **Machine-to-machine** | mTLS | Long-lived certs; no browser constraints |

> **DPoP for breadth, mTLS for depth.** Deploy DPoP as the default for all
> client types. Add mTLS for high-security internal channels where PKI exists.

## 7. GGID Implementation Strategy

### Current State

GGID issues **pure bearer tokens** — RS256 JWTs signed with RSA-2048. The
`AccessTokenClaims` struct has `tenant_id` and standard registered claims, but
no `cnf` (confirmation) claim. The gateway validates signatures via JWKS with
no proof-of-possession check.

### Gateway Middleware Flow

```
Authorization: Bearer <JWT>
DPoP: <proof-JWT>              ← DPoPVerifier (if cnf.jkt present)
X-Client-Cert: <pem>           ← MTLSCertVerifier (if cnf.x5t present)

if token has cnf claim:
  cnf.jkt → DPoPVerifier.Verify(proof, cnf.jkt, method, uri, ath)
  cnf.x5t → MTLSCertVerifier.Verify(cert, cnf.x5t)
else:
  bearer token (Phase 1: allowed, Phase 4: rejected)
```

### Go Interface Design

```go
// pkg/auth/binding/verifier.go
package binding

type BindingMethod string

const (
	BindingNone BindingMethod = ""
	BindingDPoP BindingMethod = "dpop" // RFC 9449
	BindingMTLS BindingMethod = "mtls" // RFC 8705
)

// Confirmation is the cnf claim (RFC 7800) embedded in access tokens.
type Confirmation struct {
	JKT     string `json:"jkt,omitempty"`      // JWK Thumbprint (DPoP)
	X5tS256 string `json:"x5t#S256,omitempty"` // Cert thumbprint (mTLS)
}

// VerifyResult indicates whether the sender proved key possession.
type VerifyResult struct {
	OK     bool
	Method BindingMethod
	Reason string // populated when OK == false
}

// TokenBindingVerifier is the strategy interface for all binding methods.
type TokenBindingVerifier interface {
	Method() BindingMethod
	// proof: DPoP JWT header OR client cert PEM
	// method/uri/ath: request context for replay prevention
	Verify(ctx context.Context, cnf Confirmation, proof string,
		method, uri, ath string) VerifyResult
}
```

### Verifier Implementations (Outline)

**DPoPVerifier** validates 7 checks in sequence:
1. Parse + verify DPoP proof JWT signature
2. `jwkThumbprint(header.jwk) == cnf.jkt`
3. `htm == request method`
4. `htu == request URI` (no query string)
5. `ath == sha256(accessToken)`
6. `iat` within maxSkew (default 60s)
7. `jti` not replayed (Redis-backed store, TTL = 2x maxSkew)

**MTLSCertVerifier** validates 3 checks:
1. Parse client cert from TLS peer or `X-Client-Cert` header
2. `sha256(cert.Raw) == cnf.x5t#S256`
3. Cert within validity period (`NotBefore` / `NotAfter`)

### Configuration

```yaml
# configs/gateway.yaml
token_binding:
  allowed_methods: ["bearer", "dpop", "mtls"]
  dpop:
    proof_max_age: 60s
    jti_cache_ttl: 300s
    nonce_enabled: true
  mtls:
    client_ca_path: /etc/ggid/certs/client-ca.pem
    require_client_cert: true
  # Per-client overrides in client registry:
  #   spa-console:   required_method: dpop
  #   admin-api:     required_method: mtls
  #   public-read:   required_method: bearer
```

### Integration Points

| Component | Change Required |
|---|---|
| `AccessTokenClaims` | Add `cnf *Confirmation` field |
| `TokenService.IssueAccessToken` | Accept optional binding params, embed `cnf` |
| Gateway JWT middleware | Add `TokenBindingVerifier` chain after signature check |
| Gateway TLS config | Enable `tls.RequireAnyClientCert` for mTLS |
| Redis | Add `dpop:jti:*` keyspace for nonce tracking |
| Client SDK | Add DPoP key generation + proof signing |

## 8. Migration Path

| Phase | Description | Effort | Breaking? |
|---|---|---|---|
| **0 — Current** | Pure bearer tokens, no `cnf` claim | — | — |
| **1 — DPoP optional** | Add `DPoPVerifier` + `cnf.jkt`; clients opt-in via DPoP header. Backward compatible. | ~2 wk | No |
| **2 — mTLS internal** | Gateway-to-service mTLS + `MTLSCertVerifier`. Internal API uses cert-bound tokens. | ~1 wk | No |
| **3 — Binding required** | Per-client `required_method` config. Admin/tenant/role endpoints require DPoP or mTLS. | ~1 wk | Config-driven |
| **4 — Bearer deprecated** | Remove bearer support for non-public endpoints. Public read-only APIs may keep bearer. | — | Yes (6-mo window) |

```
Phase 0 → Phase 1 → Phase 2 → Phase 3 → Phase 4
(now)     (+2 wk)   (+3 wk)   (+4 wk)   (+6 mo)
```
