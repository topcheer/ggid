# Key Management Lifecycle

## Overview

Cryptographic key management is fundamental to IAM security. This document analyzes the complete key lifecycle — generation, rotation, escrow, and destruction — and compares HSM, KMS, and software-based approaches, mapping each to GGID's implementation.

> **Related**: [Access Key Management](access-key-management.md) (1403 lines, comprehensive design)

## Key Types in GGID

| Key Type | Purpose | Algorithm | Location |
|----------|---------|-----------|----------|
| JWT signing key | Sign access/refresh tokens | RS256 / ES256 | OAuth service |
| JWT verification key | Verify token signatures | RSA public / EC public | All services |
| TLS server key | HTTPS termination | RSA 2048 / ECDSA P-256 | Gateway |
| gRPC TLS key | Inter-service mTLS | RSA 2048 | All services |
| SAML signing key | Sign SAML assertions | RSA / SHA-256 | OAuth service |
| WebAuthn RP key | Relying party identity | ES256 | Auth service |
| Password pepper | Argon2id pepper | HMAC-SHA256 | Auth service |
| Webhook HMAC key | Sign webhook payloads | HMAC-SHA256 | Audit service |
| OAuth client secret | Client authentication | Random 256-bit | OAuth service |
| API keys | Service-to-service auth | Random 256-bit | Identity service |

## Key Lifecycle Phases

### 1. Generation

| Key Type | Method | Strength |
|----------|--------|----------|
| RSA key pair | `rsa.GenerateKey(rand.Reader, 2048)` | 112-bit |
| ECDSA P-256 | `ecdsa.GenerateKey(elliptic.P256(), rand.Reader)` | 128-bit |
| Ed25519 | `ed25519.GenerateKey(rand.Reader)` | 128-bit |
| HMAC key | `crypto/rand.Read(32)` | 256-bit |
| Client secret | `crypto/rand.Read(32)` + base64url | 256-bit |

**GGID**: JWT signing key generated by `ggid-keygen` container on first run using `crypto/rand`.

```go
// RotatingKeyProvider generates initial key
func NewRotatingKeyProvider(initialKey *rsa.PrivateKey) *RotatingKeyProvider {
    return &RotatingKeyProvider{
        currentID:  generateKeyID(initialKey),
        currentKey: initialKey,
        publicKey:  &initialKey.PublicKey,
    }
}
```

### 2. Distribution

Keys must be distributed to all services that need them:

| Key | Distributed To | Method |
|-----|---------------|--------|
| JWT signing (private) | OAuth service only | Secrets manager / volume |
| JWT verification (public) | All services | JWKS endpoint (`/.well-known/jwks.json`) |
| gRPC TLS | All services | Mounted certificates |
| Password pepper | Auth service only | Environment variable / secrets manager |

**JWKS Endpoint** eliminates private key distribution — services fetch public keys dynamically:

```
Service startup → Fetch JWKS → Cache 15 min → Re-fetch on kid miss
```

### 3. Rotation

#### GGID Key Rotation

```go
// RotateKey generates a new signing key and demotes the current key
func (r *RotatingKeyProvider) RotateKey() error {
    newKey, err := rsa.GenerateKey(rand.Reader, 2048)
    if err != nil {
        return err
    }
    r.previousKey = r.currentKey  // Old key retained for grace period
    r.previousID = r.currentID
    r.currentKey = newKey
    r.currentID = generateKeyID(newKey)
    r.publicKey = &newKey.PublicKey
    r.rotationTime = time.Now()
    return nil
}
```

**Grace period**: Old key remains valid for configurable duration (default 24h) to allow existing tokens to expire naturally.

#### Rotation Schedule

| Key Type | Rotation Frequency | Grace Period |
|----------|-------------------|--------------|
| JWT signing | 90 days | 24 hours |
| TLS server | 90 days (Let's Encrypt) / 1 year (CA) | 7 days |
| gRPC mTLS | 90 days | 24 hours |
| SAML signing | 365 days | 7 days |
| HMAC webhook | 180 days | 48 hours |
| Password pepper | Never (or on breach) | N/A (requires re-hash) |
| Client secrets | On compromise | Immediate |

#### Zero-Downtime Rotation Procedure

```
T+0:   Generate new key pair
T+0:   Add new public key to JWKS (alongside old key)
T+0:   Switch signing to new key
T+0:   Publish both keys in JWKS
       ↓ (old tokens still valid, signed with old key)
T+24h: Old tokens expired
T+24h: Remove old key from JWKS
T+24h: Destroy old private key
```

### 4. Escrow and Backup

| Key Type | Backup Method | Recovery |
|----------|--------------|----------|
| JWT signing | Secrets manager (AWS SM, Vault) | Restore from backup |
| TLS | ACME / cert-manager | Re-issue via ACME |
| SAML signing | Encrypted backup to S3 | Restore + reconfigure |
| Password pepper | HSM / secrets manager | MUST NOT lose (all passwords invalid) |

**Critical**: Password pepper loss is catastrophic — all stored password hashes become unverifiable. Must have redundant escrow.

### 5. Destruction

```go
// After grace period, zero-fill and discard
func (r *RotatingKeyProvider) DestroyOldKey() {
    if r.previousKey != nil && time.Since(r.rotationTime) > r.gracePeriod {
        // Zero the key material
        for i := range r.previousKey.D {
            r.previousKey.D[i] = 0
        }
        r.previousKey = nil
        r.previousID = ""
    }
}
```

## Key Storage Backends

### HSM (Hardware Security Module)

| Feature | Rating |
|---------|--------|
| Security | Highest — keys never leave hardware |
| Performance | Medium — PKCS#11 overhead |
| Cost | High ($10K-$100K) |
| FIPS certified | Yes (FIPS 140-2 Level 3+) |
| Key extraction | Impossible |

**Use case**: Banking, government, regulated industries.

### Cloud KMS

| Provider | Service | Key Type |
|----------|---------|----------|
| AWS | KMS / CloudHSM | RSA, ECDSA, AES |
| Google | Cloud KMS | RSA, ECDSA, AES |
| Azure | Key Vault | RSA, ECDSA, AES |

| Feature | Rating |
|---------|--------|
| Security | High — keys in cloud HSM |
| Performance | High — API-based signing |
| Cost | Low ($1/key/month + per-operation) |
| Key extraction | Not possible (by design) |

**Use case**: Cloud-native deployments, SaaS.

### Software (File-Based)

| Feature | Rating |
|---------|--------|
| Security | Medium — keys on disk |
| Performance | Highest — no overhead |
| Cost | Free |
| Key extraction | Possible if disk compromised |

**Use case**: Development, testing, small deployments.

**GGID Default**: Software-based. For production, mount keys from secrets manager (Vault, AWS Secrets Manager) via Docker volume.

## GGID KMS Integration Pattern

```go
// Abstract key provider interface
type KeyProvider interface {
    PublicKey() crypto.PublicKey
    PrivateKey() crypto.PrivateKey  // nil if using KMS for signing
    KeyID() string
    Sign(data []byte) ([]byte, error)  // KMS-backed or local
}

// KMS-backed implementation
type KMSKeyProvider struct {
    keyID    string
    kmsClient *kms.Client
}

func (k *KMSKeyProvider) Sign(data []byte) ([]byte, error) {
    // Signing happens inside KMS, private key never exposed
    resp, err := k.kmsClient.Sign(ctx, &kms.SignRequest{
        KeyId:            k.keyID,
        Message:          data,
        SigningAlgorithm: "RSASSA_PKCS1_V1_5_SHA_256",
    })
    return resp.Signature, err
}
```

## Security Best Practices

1. **Never hardcode keys** in source code or Docker images
2. **Use separate keys** for each environment (dev, staging, prod)
3. **Rotate regularly** — automated rotation preferred
4. **Audit key usage** — log every signing operation
5. **Separate duties** — key generation ≠ key usage ≠ key destruction
6. **Zero-fill on destruction** — don't just delete files
7. **Test key recovery** — before you need it
8. **Monitor for key compromise** — anomalous signing patterns
