## 1. State Parameter Security Requirements

### RFC 6749 Requirements

RFC 6749 Section 10.12 (Cross-Site Request Forgery) mandates the `state`
parameter as the primary defense against CSRF in the OAuth authorization
code flow. The client generates an opaque, unpredictable value, sends it
in the authorization request, and the authorization server echoes it back
unchanged in the callback redirect. The client then verifies the returned
value matches the one it sent.

> *"The authorization server MUST NOT interact with the state parameter
> in any way other than echoing it back to the client."* — RFC 6749 §10.12

However, when GGID acts as both the **authorization server** and the
**authorization server callback handler** (server-side rendered flow), the
state parameter must be stored and validated server-side. This is because
the callback hits GGID's own redirect handler, not a third-party client.

### Minimum Security Properties

| Property | Requirement | Rationale |
|---|---|---|
| **Entropy** | Minimum 128 bits | Prevents brute-force guessing of valid state values |
| **Unpredictability** | `crypto/rand` CSPRNG | Predictable RNG (time-seeded) allows attacker to forge state |
| **Session binding** | State linked to a specific authenticated session | Prevents injection of an attacker's state into victim's session |
| **Single use** | State consumed and deleted on callback | Prevents replay attacks |
| **TTL enforced** | Maximum 10 minutes | Limits window for interception/replay |
| **Constant-time comparison** | `subtle.ConstantTimeCompare` | Prevents timing attacks on validation |

### Login CSRF Attack (When State Is Not Validated)

Login CSRF (also called "session fixation via OAuth") is the most severe
consequence of missing state validation:

```
1. Attacker initiates OAuth flow with GGID:
   GET /oauth/authorize?client_id=app&redirect_uri=https://app/callback
   &state=ATTACKER_STATE

2. GGID issues authorization code and redirects:
   https://app/callback?code=CODE_123&state=ATTACKER_STATE

3. Attacker intercepts the redirect URL (never completes it).

4. Attacker sends the URL to the victim via phishing/iframe:
   https://app/callback?code=CODE_123&state=ATTACKER_STATE

5. Victim's browser hits the callback. If state is NOT validated:
   - GGID exchanges the code for tokens
   - Tokens are bound to the ATTACKER's authorization session
   - Victim is now logged in as the attacker

6. Attacker can now read victim's activity (typed passwords, viewed data)
   because they share the same session.
```

If GGID's callback handler does not verify that the `state` value in the
redirect matches a value it previously generated and stored, this attack
succeeds silently.

### Why GGID Is Currently Vulnerable

The current `/oauth/authorize` handler in `services/oauth/internal/server/server.go`
reads `state` from the query string (line 166), passes it to
`CreateAuthorizationCode`, which only checks that it is non-empty (line 223),
and then echoes it back in the redirect URL (line 282). At no point is the
state:
- Generated by the server with cryptographic randomness
- Stored server-side with session binding
- Validated against a stored value on callback
- Constrained to single use

## 2. Entropy and Generation

### Why time.Now()-Based Generation Is Broken

GGID's existing `generateCSRFToken` pattern (in `services/gateway/internal/middleware/middleware.go`,
now patched in commit `29b51c1` but illustrative of the anti-pattern) used
`time.Now().UnixNano()` shifted per byte:

```go
// BROKEN — do NOT use
func generateCSRFToken() string {
    b := make([]byte, 16)
    t := time.Now().UnixNano()
    for i := range b {
        b[i] = byte(t >> (i * 8))
    }
    return hex.EncodeToString(b)
}
```

This produces only ~63 bits of entropy (the nanosecond timestamp), and those
bits are **fully predictable** if the attacker can estimate the server's clock
within even a few seconds. An attacker who knows the approximate generation
time can brute-force the state in under 2^30 attempts.

### Secure crypto/rand Generation

The fix requires using Go's `crypto/rand` package to generate at least 128
bits (16 bytes) of cryptographically random data:

```go
package oauth

import (
    "crypto/rand"
    "encoding/base64"
)

// generateStateParameter creates a cryptographically secure state value
// with 192 bits of entropy (24 bytes), base64url-encoded.
// This exceeds the 128-bit minimum recommended by OAuth 2.0 Security BCP §4.7.
func generateStateParameter() (string, error) {
    b := make([]byte, 24) // 192 bits of entropy
    if _, err := rand.Read(b); err != nil {
        return "", fmt.Errorf("generate state: %w", err)
    }
    return base64.RawURLEncoding.EncodeToString(b), nil
}
```

**Key properties:**
- `crypto/rand.Read` uses the OS CSPRNG (`/dev/urandom` on Linux, `SecRandomCopyBytes` on macOS).
- 24 bytes = 192 bits of entropy, well above the 128-bit minimum.
- `base64.RawURLEncoding` produces a URL-safe string (no `+` or `/`).
- Output length: 32 characters (base64url of 24 bytes).

### Nonce Generation (for OIDC)

The same secure generation applies to the OIDC `nonce` parameter:

```go
// generateNonce creates a cryptographically secure nonce for OIDC ID Token binding.
func generateNonce() (string, error) {
    b := make([]byte, 16) // 128 bits
    if _, err := rand.Read(b); err != nil {
        return "", fmt.Errorf("generate nonce: %w", err)
    }
    return base64.RawURLEncoding.EncodeToString(b), nil
}
```

## 3. Server-Side State Storage (Redis)

### Design

When GGID initiates an authorization code flow, it generates a state value,
binds it to the current session/client/redirect context, and stores it in
Redis with a short TTL. On the callback, the state is looked up, validated,
and atomically deleted (single use).

```
Redis key:   oauth:state:{state_value}
Redis value: JSON({ "session_id": "...", "client_id": "...", "redirect_uri": "...", "nonce": "..." })
TTL:         600 seconds (10 minutes)
```

### Go Implementation: Redis-Backed State Store

```go
package oauth

import (
    "context"
    "encoding/json"
    "fmt"
    "time"

    "github.com/redis/go-redis/v9"
)

// StateEntry holds the context bound to a state parameter.
type StateEntry struct {
    SessionID   string    `json:"session_id"`
    ClientID    string    `json:"client_id"`
    RedirectURI string    `json:"redirect_uri"`
    Nonce       string    `json:"nonce,omitempty"`
    CreatedAt   time.Time `json:"created_at"`
}

// StateStore manages OAuth state parameters in Redis.
type StateStore struct {
    rdb *redis.Client
    ttl time.Duration
}

// NewStateStore creates a Redis-backed state store with a 10-minute TTL.
func NewStateStore(rdb *redis.Client) *StateStore {
    return &StateStore{
        rdb: rdb,
        ttl: 10 * time.Minute,
    }
}

// Save stores the state entry with the configured TTL.
func (s *StateStore) Save(ctx context.Context, state string, entry *StateEntry) error {
    data, err := json.Marshal(entry)
    if err != nil {
        return fmt.Errorf("marshal state entry: %w", err)
    }
    key := fmt.Sprintf("oauth:state:%s", state)
    return s.rdb.Set(ctx, key, data, s.ttl).Err()
}

// ValidateAndConsume atomically retrieves and deletes the state entry.
// This ensures single-use: once consumed, the same state cannot be replayed.
// Returns ErrStateNotFound if the state does not exist or has expired.
func (s *StateStore) ValidateAndConsume(ctx context.Context, state string) (*StateEntry, error) {
    key := fmt.Sprintf("oauth:state:%s", state)

    // Lua script for atomic GETDEL — prevents race conditions where two
    // concurrent callbacks consume the same state value.
    const getDelScript = `
        local val = redis.call('GET', KEYS[1])
        if val == false then
            return nil
        end
        redis.call('DEL', KEYS[1])
        return val
    `

    result, err := s.rdb.Eval(ctx, getDelScript, []string{key}).Result()
    if err != nil {
        return nil, fmt.Errorf("state validation redis error: %w", err)
    }
    if result == nil {
        return nil, ErrStateNotFound
    }

    data, ok := result.(string)
    if !ok {
        return nil, fmt.Errorf("unexpected state value type in redis")
    }

    var entry StateEntry
    if err := json.Unmarshal([]byte(data), &entry); err != nil {
        return nil, fmt.Errorf("unmarshal state entry: %w", err)
    }

    return &entry, nil
}

var ErrStateNotFound = fmt.Errorf("state parameter not found or expired")
```

### TTL Management

- **10-minute TTL**: Matches the authorization code lifetime (currently
  10 minutes in `CreateAuthorizationCode`, line 259 of `oauth_service.go`).
- **Atomic delete**: Uses a Lua `GETDEL` script to prevent race conditions
  where two concurrent callback requests consume the same state value.
- **No cleanup job needed**: Redis auto-expires keys. No background sweep
  is required.

## 4. Signed JWT State (Alternative)

### Design

Instead of storing state in Redis, the state value itself can be a signed JWT
containing the session binding information. The server signs it on
authorization, and verifies the signature on callback.

```go
package oauth

import (
    "crypto/rand"
    "encoding/base64"
    "fmt"
    "time"

    "github.com/golang-jwt/jwt/v5"
)

// JWTStateClaims holds the session binding embedded in the state JWT.
type JWTStateClaims struct {
    SessionID   string `json:"sid"`
    ClientID    string `json:"cid"`
    RedirectURI string `json:"ruri"`
    jwt.RegisteredClaims
}

// JWTStateStore generates and validates state as a self-contained signed JWT.
type JWTStateStore struct {
    signingKey []byte // HMAC-SHA256 key
    issuer     string
    ttl        time.Duration
}

func NewJWTStateStore(signingKey []byte, issuer string) *JWTStateStore {
    return &JWTStateStore{
        signingKey: signingKey,
        issuer:     issuer,
        ttl:        10 * time.Minute,
    }
}

// Generate creates a signed JWT state value bound to the given session.
func (s *JWTStateStore) Generate(sessionID, clientID, redirectURI string) (string, error) {
    now := time.Now()
    claims := JWTStateClaims{
        SessionID:   sessionID,
        ClientID:    clientID,
        RedirectURI: redirectURI,
        RegisteredClaims: jwt.RegisteredClaims{
            Issuer:    s.issuer,
            IssuedAt:  jwt.NewNumericDate(now),
            ExpiresAt: jwt.NewNumericDate(now.Add(s.ttl)),
            Subject:   sessionID,
            ID:        generateRandomID(),
        },
    }
    token := jwt.NewWithClaims(jwt.SigningMethodHS256, claims)
    return token.SignedString(s.signingKey)
}

// Validate verifies the JWT signature and returns the embedded claims.
// Note: JWT state cannot enforce single-use without additional server-side tracking.
func (s *JWTStateStore) Validate(state string) (*JWTStateClaims, error) {
    token, err := jwt.ParseWithClaims(state, &JWTStateClaims{}, func(t *jwt.Token) (any, error) {
        if _, ok := t.Method.(*jwt.SigningMethodHMAC); !ok {
            return nil, fmt.Errorf("unexpected signing method: %v", t.Header["alg"])
        }
        return s.signingKey, nil
    })
    if err != nil {
        return nil, fmt.Errorf("invalid state token: %w", err)
    }
    claims, ok := token.Claims.(*JWTStateClaims)
    if !ok || !token.Valid {
        return nil, fmt.Errorf("invalid state claims")
    }
    return claims, nil
}

func generateRandomID() string {
    b := make([]byte, 8)
    _, _ = rand.Read(b)
    return base64.RawURLEncoding.EncodeToString(b)
}
```

### Comparison: Redis vs JWT State

| Aspect | Redis-Backed State | JWT State |
|---|---|---|
| **Single-use enforcement** | Yes (atomic GETDEL) | No (JWT is replayable without extra tracking) |
| **Server-side storage** | Required (Redis) | None (stateless) |
| **State value size** | ~32 chars (random) | ~300+ chars (JWT) |
| **Latency** | 1 Redis round-trip | 0 (signature verification only) |
| **Replay protection** | Built-in (key deleted) | Requires Redis jti blacklist |
| **Key rotation** | N/A (random values) | Must rotate HMAC key periodically |
| **Scalability** | Redis is bottleneck | Stateless, infinitely scalable |
| **Implementation complexity** | Moderate | Lower |
| **OIDC compliance** | Full | Full |
| **Recommended for GGID** | **Yes** | Alternative for read-heavy scale |

**Recommendation for GGID**: Use Redis-backed state. GGID already depends on
Redis for session management and rate limiting. The single-use guarantee from
atomic deletion is critical for security, and the additional latency (sub-
millisecond for Redis in Docker Compose) is negligible.

## 5. PKCE Interaction with State

### Complementary, Not Redundant

PKCE (Proof Key for Code Exchange, RFC 7636) and the state parameter protect
against **different** attack vectors. Both must be present for complete
authorization code flow security.

```
┌─────────────────────────────────────────────────────────────────────┐
│                  Authorization Code Flow Security Layers            │
├─────────────────────┬───────────────────────────────────────────────┤
│ Threat              │ Defense                                       │
├─────────────────────┼───────────────────────────────────────────────┤
│ Authorization code  │ PKCE (code_challenge / code_verifier)        │
│ interception        │                                               │
│ (network/MITM)      │ Attacker intercepts code but cannot          │
│                     │ exchange it without the code_verifier        │
├─────────────────────┼───────────────────────────────────────────────┤
│ CSRF / Login        │ State parameter                               │
│ fixation            │                                               │
│ (social engineering)│ Attacker injects their code into victim's    │
│                     │ session, but state mismatch is detected      │
├─────────────────────┼───────────────────────────────────────────────┤
│ ID token injection  │ OIDC nonce                                    │
│ (token substitution)│                                               │
│                     │ Attacker substitutes their ID token; nonce   │
│                     │ in token does not match request nonce         │
└─────────────────────┴───────────────────────────────────────────────┘
```

### Combined Flow Diagram

```
Client                    GGID Auth Server              Attacker
  │                            │                            │
  │── 1. Generate code_verifier, code_challenge, state ────▶│
  │   (PKCE pair + state stored in session/Redis)          │
  │                            │                            │
  │── 2. GET /authorize ───────────────────────────────────▶│
  │   ?client_id=app                                        │
  │   &redirect_uri=https://app/cb                          │
  │   &code_challenge=BASE64URL(SHA256(verifier))  [PKCE]   │
  │   &state=RANDOM_192_BITS                        [CSRF]  │
  │   &nonce=RANDOM_128_BITS                      [OIDC]    │
  │                            │                            │
  │◀── 3. 302 Redirect ─────────────────────────────────────│
  │   https://app/cb?code=AUTH_CODE&state=RANDOM_192_BITS   │
  │                            │                            │
  │                            │   4. Attacker tries to     │
  │                            │◀── inject their code with  │
  │                            │    different state         │
  │                            │                            │
  │                            │   5. State mismatch →      │
  │                            │──▶ REJECT (CSRF blocked)   │
  │                            │                            │
  │── 6. POST /token ───────────────────────────────────────▶│
  │   code=AUTH_CODE                                        │
  │   code_verifier=ORIGINAL_VERIFIER              [PKCE]    │
  │                            │                            │
  │                            │   7. Verify SHA256(verifier)│
  │                            │   == stored challenge      │
  │                            │   (interception blocked)   │
  │                            │                            │
  │◀── 8. access_token + id_token ──────────────────────────│
  │   id_token contains nonce=RANDOM_128_BITS     [OIDC]    │
  │                            │                            │
  │── 9. Verify nonce in id_token ──▶ matches → ACCEPT      │
```

### PKCE in GGID Today

GGID already implements PKCE in the `AuthorizationCode` domain model
(`services/oauth/internal/domain/models.go`, lines 108-138) with S256 support.
The `CreateAuthorizationCode` method enforces PKCE for public clients
(`RequiresPKCE()`, line 233). However, state validation is completely missing,
creating a gap where code interception is defended but CSRF is not.

## 6. OIDC nonce Binding

### Purpose

The OIDC `nonce` parameter (OpenID Connect Core §3.1.2.1) prevents ID token
injection and replay attacks. Unlike `state` (which protects the redirect),
`nonce` is embedded in the issued ID token and must be verified after token
exchange.

### Flow

```
1. Authorization Request:
   /authorize?...&nonce=ABC123

2. Authorization Code stored with nonce.

3. Token Exchange returns id_token with "nonce":"ABC123" claim.

4. Client verifies: id_token.nonce == stored nonce from step 1.
   If mismatch → reject the ID token (possible injection).
```

### Binding nonce to state

For maximum security, the `nonce` should be stored alongside the `state` in
the Redis state entry. This creates a single atomic record that ties together:
- The user session (CSRF protection)
- The authorization request parameters (redirect_uri, client_id)
- The nonce for ID token verification (token injection protection)

### Go Implementation: Nonce Generation and Verification

```go
// generateNonce creates a 128-bit nonce for OIDC ID Token binding.
func generateNonce() (string, error) {
    b := make([]byte, 16)
    if _, err := rand.Read(b); err != nil {
        return "", fmt.Errorf("generate nonce: %w", err)
    }
    return base64.RawURLEncoding.EncodeToString(b), nil
}

// verifyNonce checks that the nonce in the ID token matches the nonce
// stored in the state entry. Uses constant-time comparison.
func verifyNonce(idTokenNonce string, expectedNonce string) error {
    if idTokenNonce == "" {
        return fmt.Errorf("id_token missing nonce claim")
    }
    if subtle.ConstantTimeCompare([]byte(idTokenNonce), []byte(expectedNonce)) != 1 {
        return fmt.Errorf("nonce mismatch — possible ID token injection")
    }
    return nil
}

// verifyIDTokenNonce extracts the nonce from a JWT ID token and validates it.
func verifyIDTokenNonce(idToken string, expectedNonce string) error {
    // Parse the ID token to extract the nonce claim.
    token, _, err := jwt.NewParser().ParseUnverified(idToken, jwt.MapClaims{})
    if err != nil {
        return fmt.Errorf("parse id_token: %w", err)
    }
    claims, ok := token.Claims.(jwt.MapClaims)
    if !ok {
        return fmt.Errorf("invalid id_token claims")
    }

    nonce, ok := claims["nonce"].(string)
    if !ok {
        return fmt.Errorf("id_token missing nonce claim")
    }

    return verifyNonce(nonce, expectedNonce)
}
```

### Current GGID nonce Handling

GGID already stores the nonce in the `AuthorizationCode` struct (line 258) and
injects it into the ID token during `issueIDToken` (line 455). The missing piece
is **client-side verification**: the code that exchanges the authorization code
for tokens must verify the nonce in the returned ID token against the nonce
it sent in the authorization request.

## 7. State Validation Flow

### Complete Callback Handler

This is the full callback handler flow that GGID should implement. It combines
state validation, single-use enforcement, redirect_uri matching, and PKCE
verification:

```go
// HandleOAuthCallback processes the redirect from the authorization endpoint.
// It validates the state, consumes it (single use), and exchanges the code.
func HandleOAuthCallback(w http.ResponseWriter, r *http.Request) {
    if r.Method != http.MethodGet {
        writeJSON(w, http.StatusMethodNotAllowed, map[string]string{"error": "method_not_allowed"})
        return
    }

    code := r.URL.Query().Get("code")
    state := r.URL.Query().Get("state")
    errorParam := r.URL.Query().Get("error")

    // 1. Check for OAuth error response from authorization server.
    if errorParam != "" {
        writeJSON(w, http.StatusBadRequest, map[string]string{
            "error":             errorParam,
            "error_description": r.URL.Query().Get("error_description"),
        })
        return
    }

    // 2. Both code and state are required.
    if code == "" || state == "" {
        writeJSON(w, http.StatusBadRequest, map[string]string{
            "error":             "invalid_request",
            "error_description": "code and state parameters are required",
        })
        return
    }

    // 3. Validate and consume the state (single-use, atomic).
    stateEntry, err := stateStore.ValidateAndConsume(r.Context(), state)
    if err != nil {
        // State not found → expired, already used, or never issued by us.
        // This is the CSRF defense: an attacker's state will not be in our store.
        writeJSON(w, http.StatusBadRequest, map[string]string{
            "error":             "invalid_state",
            "error_description": "state parameter is invalid, expired, or already used",
        })
        return
    }

    // 4. Verify session binding: the callback must come from the same session
    // that initiated the authorization request.
    sessionCookie, err := r.Cookie("ggid_session")
    if err != nil || sessionCookie.Value == "" {
        writeJSON(w, http.StatusUnauthorized, map[string]string{
            "error":             "invalid_session",
            "error_description": "session cookie required for state validation",
        })
        return
    }
    if subtle.ConstantTimeCompare([]byte(sessionCookie.Value), []byte(stateEntry.SessionID)) != 1 {
        writeJSON(w, http.StatusForbidden, map[string]string{
            "error":             "session_mismatch",
            "error_description": "state was issued to a different session",
        })
        return
    }

    // 5. Verify redirect_uri matches the one stored in the state entry.
    redirectURI := r.URL.Query().Get("redirect_uri")
    if redirectURI != "" && redirectURI != stateEntry.RedirectURI {
        writeJSON(w, http.StatusBadRequest, map[string]string{
            "error":             "redirect_uri_mismatch",
            "error_description": "redirect_uri does not match the original request",
        })
        return
    }

    // 6. Exchange the authorization code for tokens.
    tokenResp, err := oauthSvc.ExchangeAuthorizationCode(r.Context(), &service.TokenExchangeRequest{
        TenantID:     stateEntry.TenantID,
        Code:         code,
        RedirectURI:  stateEntry.RedirectURI,
        ClientID:     stateEntry.ClientID,
        CodeVerifier: getPKCEVerifierFromSession(sessionCookie.Value),
    })
    if err != nil {
        writeJSON(w, http.StatusBadRequest, map[string]string{
            "error":             "token_exchange_failed",
            "error_description": err.Error(),
        })
        return
    }

    // 7. If OIDC flow, verify nonce in ID token.
    if tokenResp.IDToken != "" && stateEntry.Nonce != "" {
        if err := verifyIDTokenNonce(tokenResp.IDToken, stateEntry.Nonce); err != nil {
            writeJSON(w, http.StatusBadRequest, map[string]string{
                "error":             "nonce_mismatch",
                "error_description": "ID token nonce verification failed",
            })
            return
        }
    }

    // 8. Success — return tokens to the client.
    writeJSON(w, http.StatusOK, tokenResp)
}
```

### Key Security Checks in Order

1. **OAuth error** → propagate to client
2. **Missing code/state** → reject immediately
3. **State not in store** → reject (CSRF defense triggers here)
4. **Session mismatch** → reject (login CSRF defense)
5. **redirect_uri mismatch** → reject (open redirect defense)
6. **Token exchange** → delegates to existing `ExchangeAuthorizationCode`
7. **Nonce mismatch** → reject (ID token injection defense)

## 8. GGID State Validation P0 Fix Design

### Current State Handling (Vulnerability Analysis)

**File: `services/oauth/internal/server/server.go`**

The `/oauth/authorize` handler (lines 157-290):

```go
// Line 166: state is READ from the query string
state := r.URL.Query().Get("state")

// Line 256-267: state is PASSED THROUGH to CreateAuthorizationCode
code, err := oauthSvc.CreateAuthorizationCode(ctx, &service.AuthorizeRequest{
    ...
    State: state,    // ← passed through, never stored
    ...
})

// Line 280-282: state is ECHOED BACK in the redirect URL
if state != "" {
    redirectURL += "&state=" + state
}
```

**File: `services/oauth/internal/service/oauth_service.go`**

The `CreateAuthorizationCode` method (lines 202-267):

```go
// Line 222-225: state is only checked for NON-EMPTY, never stored or validated
if req.State == "" {
    return "", errors.InvalidArgument("state parameter is required")
}
```

**Critical gap**: There is no callback handler. The `/oauth/authorize` endpoint
returns a JSON response with the redirect URL and code. There is no
`/oauth/callback` endpoint that would validate the state. The token exchange
endpoint (`/oauth/token`) does not check state at all — it goes directly to
code exchange.

### Fix Architecture

```
┌─────────────────────────────────────────────────────────┐

func (s *OAuthService) GenerateAndStoreState(ctx context.Context, sessionID, clientID, redirectURI, nonce string) (string, error) {
    state, err := generateStateParameter()
    if err != nil {
        return "", errors.Internal("generate state", err)
    }
    if s.stateStore != nil {
        entry := &StateEntry{
            SessionID:   sessionID,
            ClientID:    clientID,
            RedirectURI: redirectURI,
            Nonce:       nonce,
            CreatedAt:   time.Now(),
        }
        if err := s.stateStore.Save(ctx, state, entry); err != nil {
            return "", errors.Internal("store state", err)
        }
    }
    return state, nil
}
```

### Risk Assessment

| Risk | Mitigation |
|---|---|
| Redis unavailable in dev/test | `StateStore` is nil-safe: if `rdb == nil`, state generation falls back to random-only (no storage). Tests should run with embedded miniredis. |
| Breaking API change (constructor signature) | All callers updated in same PR. Use functional options pattern if backward compat needed. |
| Existing tests use hardcoded `"test-state"` | Tests must be updated to use `GenerateAndStoreState()` or mock the StateStore. |
| PAR flow also passes state through | PAR (`par.go:25`) stores state in `PushedAuthorizationRequest` — needs same treatment (generate server-side, store in StateStore). |

### Testing Strategy

1. **Unit tests**: `StateStore.Save`, `ValidateAndConsume`, atomicity (concurrent access)
2. **Integration tests**: Full authorize → callback → token exchange flow with state validation
3. **Security tests**: Replay attack, cross-session attack, expired state, missing state
4. **Negative tests**: State from different session, state after TTL, state consumed twice

### Acceptance Criteria

- [ ] `/oauth/authorize` generates state with `crypto/rand` (192 bits minimum)
- [ ] State is stored in Redis with session binding and 10-minute TTL
- [ ] `/oauth/callback` endpoint exists and validates state atomically
- [ ] State is single-use (second validation fails)
- [ ] Session binding enforced (cross-session state rejected)
- [ ] nonce verified in ID token after token exchange
- [ ] All existing tests pass with updated constructor
- [ ] New security tests pass (replay, cross-session, expiry)
- [ ] `go build ./...` and `go vet ./...` pass clean
