# WebAuthn Attestation Chain: Trust Models and Certificate Validation

> **Scope:** Deep dive into each attestation format's certificate chain structure,
> trust model, and verification process. For authenticator matrix and AAGUID tables,
> see [fido-alliance-interop-report.md](./fido-alliance-interop-report.md).

---

## 1. Overview

WebAuthn attestation is a cryptographic proof that a credential was generated by a
genuine hardware or software authenticator — not by malware or a credential harvester.
The authenticator signs its own attestation key during manufacturing, and that signature
chains to a root CA owned by the device vendor (Yubico, Google, Microsoft, Apple, etc.).

The attestation **format** (`fmt` field in `attStmt`) determines:

- What cryptographic material is presented (raw signature, X.509 cert chain, JWS)
- How the signature is computed (what bytes are signed)
- Which root CA the chain must validate against
- How much privacy information leaks about the device

Seven formats are defined in [WebAuthn Level 2](https://www.w3.org/TR/webauthn-2/):
`none`, `packed`, `tpm`, `android-key`, `android-safetynet`, `fido-u2f`, `apple`.

The Relying Party (RP) controls which formats it is willing to accept via the
**attestation conveyance preference** sent in `navigator.credentials.create()`.

---

## 2. Attestation Conveyance Preferences

| Preference | Behavior | Use Case |
|---|---|---|
| `"none"` | RP explicitly opts out; authenticator sends `fmt: "none"` | Consumer apps, privacy-first |
| `"indirect"` | RP wants attestation but permits anonymization (CA proxy re-signs) | Balanced privacy + assurance |
| `"direct"` | RP demands the authenticator's raw attestation certificate | Enterprise, compliance |
| `"enterprise"` | RP filters by AAGUID — only specific authenticator models accepted | Government, finance, regulated |

**Privacy vs. security tradeoff:**
Attestation certificates are globally unique per device. If the same certificate is
presented across multiple RPs, those RPs (or anyone observing) can correlate the
user's device across services — a fingerprinting vector. The `none` preference
eliminates this risk entirely; `direct` maximizes it.

**Recommendation:** Use `"none"` for consumer applications (the default in most
browsers and passkey implementations). Use `"direct"` only in enterprise or regulated
environments where device attestation is a compliance requirement, and accept the
privacy tradeoff explicitly.

## 3. Format Deep Dives

### none

```
attStmt = {}   // empty map
```

- **fmt:** `"none"`
- **Data:** No attestation signature or certificate. The `attStmt` map is empty.
- **AAGUID:** Still present in `authenticatorData` — identifies the authenticator model
  for RP-side logging/analytics, but carries no cryptographic proof.
- **Trust model:** No verifiable claim. The RP trusts the credential on the basis of
  user verification alone (biometric, PIN).
- **Verification:** Accept unconditionally. Nothing to verify.
- **Go:** No action. The go-webauthn library returns `AttestationType: "None"`.

### packed

- **fmt:** `"packed"`
- **Used by:** Hardware security keys (YubiKey 5 Series, Google Titan, Feitian ePass).
  The most common attestation for FIDO2 authenticators.
- **attStmt:** `{ alg: int, sig: bytes, x5c: [cert...] }` or `{ alg, sig, ecdaaKeyId }`
- **Signature scope:** `authenticatorData || SHA-256(clientDataJSON)` — signed by the
  private key corresponding to the leaf certificate's public key.
- **Trust chain:**
  ```
  Authenticator attestation key (leaf)
      └─ signed by ─> Manufacturer intermediate CA
                         └─ signed by ─> Manufacturer root CA (in FIDO MDS)
  ```
- **Verification:**
  1. Extract leaf cert from `x5c[0]`.
  2. Verify `sig` over `authData || clientDataHash` using leaf cert public key.
  3. Build cert chain from `x5c` and validate against FIDO MDS root for this AAGUID.
  4. Check EKU includes `id-fido-gen-ce-aaguid` (OID 1.3.6.1.4.1.45724.1.1.4).
- **Go:**
  ```go
  // Signature verification (library handles this):
  // verifyPackedSignature(attStmt.Sig, signedData, leafCertPubKey)

  // Cert chain validation (GGID must add):
  roots := getFIDORootCerts(aaguid)
  pool := x509.NewCertPool()
  for _, c := range roots { pool.AddCert(c) }
  leaf, _ := x509.ParseCertificate(x5c[0])
  _, err := leaf.Verify(x509.VerifyOptions{
      Roots:       pool,
      KeyUsages:   []x509.ExtKeyUsage{x509.ExtKeyUsageUnknown},
  })
  ```

### tpm

- **fmt:** `"tpm"`
- **Used by:** Windows Hello (TPM 2.0), platform authenticators on Windows laptops.
- **attStmt:** `{ ver: "2.0", alg: int, sig: bytes, x5c: [cert...],
  info: tpm2b_name, pubArea: bytes, extraData: bytes }`
- **Complexity:** HIGH. Requires parsing TPM 2.0 data structures (`TPM2B_PUBLIC`,
  `TPMT_PUBLIC`, `TPMT_SIGNATURE`) per the [TPM 2.0 spec](https://trustedcomputinggroup.org/resource/tpm-library-specification/).
- **Trust chain:**
  ```
  TPM Endorsement Key (EK) cert (leaf)
      └─ signed by ─> TPM vendor CA (Infineon, STMicro, Nuvoton)
                         └─ signed by ─> Microsoft attestation root
  ```
- **Verification:**
  1. Parse `pubArea` to extract the TPM public key (COSE-equivalent).
  2. Reconstruct `extraData` = `authData || clientDataHash` and compare to attStmt.
  3. Verify `sig` as a TPM signature (`TPMT_SIGNATURE`) over `extraData`.
  4. Validate EK certificate chain to TPM vendor root.
  5. Check that `pubArea` key matches the credential public key.
- **Go:** No standard TPM 2.0 parsing library in Go's stdlib. Requires either
  `github.com/google/go-tpm` or manual CBOR/TPM structure parsing. This is the
  most labor-intensive format to implement correctly.

### android-key

- **fmt:** `"android-key"`
- **Used by:** Android 7.0+ devices with hardware-backed keystore (Pixel, Samsung, etc.)
- **attStmt:** `{ alg: int, sig: bytes, x5c: [cert...] }`
- **Signature scope:** `authenticatorData || SHA-256(clientDataJSON)` — same as packed.
- **Trust chain:**
  ```
  Device keymaster cert (leaf, in Android Keystore)
      └─ signed by ─> Google Hardware Attestation intermediate
                         └─ signed by ─> Google Hardware Attestation Root
  ```
- **Verification:**
  1. Verify `sig` using leaf cert public key.
  2. Validate cert chain to Google Hardware Attestation root CA.
  3. Parse keymaster extension **OID 1.3.6.1.4.1.11129.2.1.17** — contains
     attestation challenge, application ID, keymaster version, security level
     (`software` vs `trustedEnvironment` vs `strongBox`).
  4. Verify the attestation challenge in the extension matches `clientDataHash`.
- **Go:**
  ```go
  ext := leaf.Extensions
  keymasterExt := ext.GetById(asn1.ObjectIdentifier{1, 3, 6, 1, 4, 1, 11129, 2, 1, 17})
  // Parse KeyDescription sequence: attestationChallenge, applicationId, ...
  ```

### android-safetynet

- **fmt:** `"android-safetynet"`
- **Status:** DEPRECATED. Google is replacing SafetyNet with the **Play Integrity API**.
  Older Android devices (<7.0 without keymaster) still use this format.
- **attStmt:** `{ ver: string, response: JWS-string }`
- **JWS payload contains:** `nonce` (must equal `SHA-256(authData || clientDataHash)`),
  `timestampMs`, `ctsProfileMatch`, `basicIntegrity`, `apkPackageName`.
- **Trust model:** Google signs the JWS with the Google attestation signing key.
  No cert chain in the traditional sense — verify the JWS signature directly.
- **Verification:**
  1. Parse JWS, extract header `x5c` (Google's attestation cert chain).
  2. Verify JWS signature using leaf cert public key.
  3. Validate cert chain to Google root.
  4. Check `nonce` matches `SHA-256(authData || clientDataHash)`.
  5. Check `timestampMs` is within acceptable freshness window (e.g. < 60s old).
  6. Check `ctsProfileMatch == true` (device passed Compatibility Test Suite).
- **GGID policy:** Accept for backward compatibility, log a deprecation warning.
  Plan migration path to `android-key` + Play Integrity API.

### fido-u2f

- **fmt:** `"fido-u2f"`
- **Used by:** Legacy U2F security keys (pre-FIDO2 YubiKey 4, FIDO U2F-only devices).
- **attStmt:** `{ x5c: [cert...], sig: bytes }`
- **Signature scope:** Unique pre-image:
  `0x00 || rpIdHash || clientDataHash || credentialId || credentialPublicKey(COSE)`
  This differs from packed — the U2F format signs a different byte sequence.
- **Trust chain:**
  ```
  U2F attestation cert (leaf, per-device or batch)
      └─ signed by ─> Manufacturer U2F attestation CA
  ```
- **Verification:**
  1. Extract leaf cert from `x5c[0]`.
  2. Reconstruct the signed bytes: `0x00 || rpIdHash || clientDataHash || credId || pubKey`.
  3. Verify `sig` using leaf cert public key.
  4. Validate cert chain to manufacturer root (Yubico U2F root, Feitian root, etc.).
  5. Verify credential public key format is uncompressed P-256 point.
- **GGID policy:** Accept for backward compatibility with older security keys.
  Most modern keys present `packed` attestation, so `fido-u2f` is increasingly rare.

### apple

- **fmt:** `"apple"`
- **Used by:** iOS 13.3+ and macOS with Touch ID / Face ID (platform authenticators).
- **attStmt:** `{ x5c: [cert...] }` — note: NO `sig` field.
- **Key insight:** The signature proving authenticator genuineness is not in `attStmt`.
  It is embedded in the leaf certificate itself as a custom extension. Apple's
  **Anonymous Attestation CA** re-signs the attestation to break device correlation.
- **Trust chain:**
  ```
  Apple Anonymous Attestation cert (leaf, unique per registration)
      └─ signed by ─> Apple Anonymous Attestation CA
                         └─ signed by ─> Apple Attestation Root CA
  ```
- **Verification:**
  1. Parse `x5c` chain (typically leaf + intermediate).
  2. Validate chain to Apple Attestation Root CA.
  3. Extract nonce from leaf cert extension **OID 1.2.840.113635.100.8.2**
     (Apple attestation extension, ASN.1 DER encoded).
  4. Verify nonce equals `SHA-256(authenticatorData || clientDataHash)`.
- **Privacy:** High. Each registration produces a unique, unlinkable certificate.
  Apple's anonymization CA ensures the same device cannot be tracked across RPs.
- **Go:**
  ```go
  appleOID := asn1.ObjectIdentifier{1, 2, 840, 113635, 100, 8, 2}
  ext := leaf.Extensions.GetById(appleOID)
  nonce := parseAppleNonce(ext.Value)
  expectedNonce := sha256.Sum256(append(authData, clientDataHash...))
  if !bytes.Equal(nonce, expectedNonce[:]) {
      return errors.New("apple attestation nonce mismatch")
  }
  ```

## 4. Certificate Chain Validation Process

### General Steps (all formats with x5c)

```
Step 1: Extract x5c[0] as leaf certificate
Step 2: Build intermediate chain from x5c[1..n]
Step 3: Load root CAs from FIDO MDS metadata for this AAGUID
Step 4: x509.Verify with custom CertPool
Step 5: Check: notExpired, digitalSignature keyUsage, EKU allows webauthn
Step 6: (Optional) Check revocation via CRL/OCSP — rarely done for FIDO
Step 7: Extract leaf cert public key for attestation signature verification
```

### FIDO MDS Root Loading

Root certificates are NOT shipped with the authenticator. The RP must fetch them
from the **FIDO Metadata Service (MDS)** — a signed BLOB at
`https://mds.fidoalliance.org/` containing metadata statements for every certified
authenticator, each with its `attestationRootCertificates` array.

```go
func getFIDORootCerts(aaguid uuid.UUID) []*x509.Certificate {
    mdsEntry := fidoMDS.Lookup(aaguid) // cached MDS BLOB
    var roots []*x509.Certificate
    for _, certPEM := range mdsEntry.AttestationRootCertificates {
        block, _ := pem.Decode([]byte(certPEM))
        cert, _ := x509.ParseCertificate(block.Bytes)
        roots = append(roots, cert)
    }
    return roots
}
```

### Full Verification Function

```go
// VerifyAttestationChain validates an attestation cert chain to FIDO roots.
func VerifyAttestationChain(x5c [][]byte, aaguid uuid.UUID) error {
    if len(x5c) == 0 {
        return errors.New("empty x5c")
    }
    leaf, err := x509.ParseCertificate(x5c[0])
    if err != nil {
        return fmt.Errorf("parse leaf cert: %w", err)
    }

    // Build root pool from FIDO MDS.
    roots := getFIDORootCerts(aaguid)
    if len(roots) == 0 {
        return fmt.Errorf("no root certs for AAGUID %s", aaguid)
    }
    rootPool := x509.NewCertPool()
    for _, c := range roots {
        rootPool.AddCert(c)
    }

    // Build intermediate pool from x5c[1..n-1].
    interPool := x509.NewCertPool()
    for _, raw := range x5c[1:] {
        if cert, err := x509.ParseCertificate(raw); err == nil {
            interPool.AddCert(cert)
        }
    }

    _, err = leaf.Verify(x509.VerifyOptions{
        Roots:         rootPool,
        Intermediates: interPool,
        KeyUsages:     []x509.ExtKeyUsage{x509.ExtKeyUsageUnknown},
        CurrentTime:   time.Now(),
    })
    return err
}
```

## 5. Trust Model Summary

| Format | Signer | Root CA | Complexity | Privacy | GGID Default |
|---|---|---|---|---|---|
| `none` | N/A | N/A | None | Maximum | **Accepted** |
| `packed` | Authenticator key | Manufacturer root (FIDO MDS) | Low | Low (device-identifiable) | Accepted |
| `tpm` | TPM EK cert | Microsoft / TPM vendor root | Very High | Low (device-identifiable) | Accepted (no chain check) |
| `android-key` | Keystore key | Google Hardware Attestation root | Medium | Low (device-identifiable) | Accepted (no chain check) |
| `android-safetynet` | Google JWS | Google root | Medium | Medium (app-level) | Accepted (deprecated) |
| `fido-u2f` | U2F key | Manufacturer U2F root | Low | Low (batch-identifiable) | Accepted |
| `apple` | Apple anon cert | Apple Attestation root | Medium | High (per-registration unique) | Accepted (no chain check) |

**Key takeaway:** The go-webauthn library verifies the *attestation signature*
(cryptographic proof that the data was signed by the device key), but does NOT
validate the *certificate chain to a trusted root* unless the RP provides root CAs.
Without chain validation, the RP accepts attestation from any source that produces
a well-formed signature — including a software emulator pretending to be a hardware key.

## 6. GGID Implementation Status

Examining `services/auth/internal/webauthn/handler.go`:

**Current state:**
- Uses `go-webauthn/webauthn` library for registration and authentication.
- `webauthn.Config` is created with `RPDisplayName`, `RPID`, `RPOrigins` — **no
  `AttestationPreference` or `RootCerts` configured**.
- `BeginRegistration` calls the library without passing attestation conveyance
  preference — defaults to `"none"` (browser may still send attestation).
- `CreateCredential` (line 535) performs library-level attestation verification:
  validates the signature format, checks `authenticatorData` flags, verifies
  client data JSON origin and challenge. But does **not** validate cert chains
  to FIDO MDS roots.
- `Credential` struct (line 36) stores `AttestationType` and `AAGUID` — useful
  for logging and analytics, but no policy enforcement based on these fields.

**What's missing:**
1. FIDO MDS integration — no metadata BLOB fetch, no root CA cache.
2. Per-format cert chain validation to vendor roots.
3. Per-tenant attestation policy (which formats are trusted).
4. AAGUID allowlist/denylist for enterprise mode.
5. Revocation checking (CRL/OCSP).

**Current effective policy:** Accept all attestation formats. The library verifies
that the credential creation response is cryptographically well-formed and the
signature is valid, but any device (including emulators) that produces a valid
signature is accepted. This is adequate for consumer passkey flows where the RP
relies on user verification (biometric/PIN) rather than device attestation.

## 7. Roadmap

### Phase 1: Observability (2-3 days)
- Log `AttestationType` and `AAGUID` for every registration.
- Build a dashboard showing authenticator distribution across tenants.
- No behavior change — just telemetry.

### Phase 2: packed + apple Chain Validation (3-4 days)
- Implement FIDO MDS BLOB fetcher with signature verification.
- Cache root certs per AAGUID in Redis.
- Add `VerifyAttestationChain()` for `packed` and `apple` formats.
- Make it opt-in per tenant (feature flag) — fail closed or fail open configurable.

### Phase 3: tpm + android-key Validation (1-2 weeks)
- Add TPM 2.0 structure parsing (pubArea, extraData, signature).
- Parse Android keymaster extension OID.
- Validate chains to Microsoft and Google roots.
- Most complex phase — consider using `go-tpm` library.

### Phase 4: Per-Tenant Attestation Policy (3-5 days)
- Tenant config: `allowed_attestation_formats: ["packed", "apple", "none"]`.
- Tenant config: `allowed_aaguids: [...]` for enterprise mode.
- Enforcement in `finishRegistration` before persisting credential.

### Phase 5: android-safetynet Deprecation + Play Integrity (ongoing)
- Log deprecation warnings for `android-safetynet` registrations.
- Add Play Integrity API token verification as a new attestation path.
- Timeline: follow Google's deprecation schedule (no hard deadline yet).

**Total estimated effort:** Phase 1-2 (~1 week), Phase 3 (~2 weeks), Phase 4-5 (~1 week).
